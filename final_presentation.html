System Overview

A modular monolith implementing a city parking reservation platform. Modules communicate through well-defined public APIs and Spring's application-event bus — preventing direct cross-module coupling.

🏗️

Architecture Style

Modular monolith with Spring Modulith — strict module boundaries enforced at test time.

📦

7 Domain Modules

⚡

Event-Driven

🔒

Concurrency Safe

Pessimistic DB locks (SELECT FOR UPDATE) prevent double-booking under concurrent load.

REST API

Module Responsibilities

Each module owns its data, its public API interfaces, and its event contracts.

Module Responsibility Communication
Reservation Create / cancel time-slot bookings; search available spaces; prevent double-booking via pessimistic lock. publishes ReservationCreatedEvent publishes ReservationCancelledEvent uses Pricing uses ZoneManagement
Billing Generate and settle invoices for reservations and EV charging sessions. listens to ReservationCreatedEvent listens to ChargingCompletedEvent publishes InvoiceGeneratedEvent uses Pricing
Charging Manage EV charging sessions (start / stop); integrate with reservation lifecycle. listens to ReservationCancelledEvent publishes ChargingStartedEvent publishes ChargingCompletedEvent uses Reservation
Notification Persist user notifications for all relevant domain events. listens to ReservationCreatedEvent listens to ReservationCancelledEvent listens to InvoiceGeneratedEvent listens to UserRegisteredEvent listens to ZoneCreatedEvent
Pricing Provide per-zone, per-space-type pricing rules and fee estimates. uses ZoneManagement
User Management Register users, issue JWT tokens, enforce role-based access control. publishes UserRegisteredEvent owns User aggregate
Zone Management Manage parking zones and spaces; track physical occupancy state. publishes ZoneCreatedEvent owns Zone & Space aggregates

Interaction Patterns

Two distinct coupling modes keep modules independent.

Direct API Call (uses)

Synchronous in-process call through a module's public interface (e.g. IPricingPolicy, ISpaceQuery). The caller depends on the interface, not the implementation.

Event Subscription (listens to)

Async @ApplicationModuleListener — fires after the publishing transaction commits in a new transaction. No direct compile-time dependency from publisher to subscriber.

Pessimistic Locking

SELECT FOR UPDATE on the space row serializes concurrent reservation requests — exactly one thread wins the lock and proceeds; others receive HTTP 409 Conflict.

Physical vs. Logical State

Space state (FREE / RESERVED / OCCUPIED) tracks physical presence only. Time-slot availability is computed on-demand via overlap queries — never persisted as state.
